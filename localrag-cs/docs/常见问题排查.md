## 📊 服务状态异常排查

### 问题1: test_services.sh 显示某个服务 ❌ 异常

#### 排查步骤：
```bash
# 1. 检查进程是否存在
ps aux | grep llama-server
ps aux | grep uvicorn

# 2. 检查端口是否被占用
netstat -tlnp | grep -E '8000|8080|8081|8082|9200'

# 3. 查看对应日志
# LLM日志:      tail -n 100 /home/zbs/llama_serve.log
# Embedding日志: tail -n 100 /home/zbs/llama_serve_8081.log
# Reranker日志: tail -n 100 /home/zbs/llama_serve_8082.log
# API日志:     tail -n 100 logs/api.log
```

#### 常见原因：
- 端口被其他进程占用 → 杀干净再重启
- 显存不够导致模型加载失败 → 关闭其他占用显存的程序
- 配置文件路径错误 → 检查config.py中的地址配置

## 🔧 API 服务问题

### 问题2: API 服务启动失败，报错 "Address already in use"

**原因：** 端口 8000 被占用

**解决：**
```bash
# 找到占用端口的进程
lsof -ti :8000

# 杀掉进程
kill -9 <PID>

# 或者修改 app.py 中的端口号
# uvicorn.run(app, host='0.0.0.0', port=8001)
```

---

### 问题3: API 接口返回 503 Service Unavailable

**原因：** 依赖服务不可用

**排查：**
```bash
# 1. 先检查哪个服务挂了
bash scripts/test_services.sh

# 2. 看具体报错信息
# 如果是 ES 异常：检查 ES 是否启动，地址密码是否正确
# 如果是 Embedding 异常：检查 llama.cpp 服务是否在 8081 端口
# 如果是 LLM 异常：检查 llama.cpp 服务是否在 8080 端口
```

## 🧠 模型服务问题

### 问题4: llama.cpp 服务启动失败，报 CUDA error

**原因：** GPU显存不足

**解决：**
```bash
# 1. 查看显存占用
nvidia-smi

# 2. 杀其他占用显存的程序
# 比如浏览器、其他模型服务等

# 3. 减少 --n-gpu-layers 层数
# 编辑 /home/zbs/llama_serve.sh
# 把 --n-gpu-layers 99 改成 --n-gpu-layers 50
```

---

### 问题5: 模型服务启动了，但请求超时

**原因：** 模型还在加载中，或者网络问题

**排查：**
```bash
# 1. 查看模型是否加载完成
curl http://localhost:8080/health

# 2. 如果返回 503，说明还在加载，等1-2分钟再试

# 3. 如果一直加载失败，看日志
tail -f /home/zbs/llama_serve.log
```

## 🔍 检索功能问题

### 问题6: 检索结果为空，或者完全不相关

**排查步骤：**
1. ✅ ES 索引是否存在？数据是否正确灌入？
2. ✅ 向量字段是不是空？检查 embedding 字段是否有值
3. ✅ 查询重写是不是把问题改得面目全非？
4. ✅ BM25 检索能不能搜到？向量检索能不能搜到？

**命令：**
```bash
# 检查索引是否存在
curl -k -u elastic:密码 "https://192.168.1.3:9200/_cat/indices?v"

# 检查索引数据量
curl -k -u elastic:密码 "https://192.168.1.3:9200/cs_knowledge_base/_count"

# 直接测试 BM25 检索
curl -k -u elastic:密码 -X POST "https://192.168.1.3:9200/cs_knowledge_base/_search" \
  -H "Content-Type: application/json" \
  -d '{"query":{"match":{"question":"工伤"}},"size":3}'
```

### 问题7: 混合检索只有 BM25 结果，没有向量结果

**原因：** 向量检索通道有问题

**排查：**
```bash
# 1. 测试 Embedding 服务是否正常

  -H "Content-Type: application/json" \
  -d '{"input": "测试问题"}'

# 2. 检查文档的 embedding 字段是否正确
# 维度是不是 1024 维，格式是不是数组
```

## 🤖 LLM 生成问题

### 问题8: 回答很慢，十几秒才返回

**正常现象：** 7B模型在消费级GPU上就是这个速度

**优化方法：**
1. 升级更好的显卡（RTX 4090 比 3090 快2倍）
2. 用更小的模型（Qwen-1.8B 快很多）
3. 减少 max_tokens 参数（config.py 中修改）

---

### 问题9: 回答胡说八道，不按知识库内容回答

**原因排查：**
1. 🔴 知识库没有相关内容 → 扩充知识库
2. 🟡 检索到的文档不相关 → 优化检索效果
3. 🟠 重排后Top3不对 → 优化Reranker参数
4. 🟢 LLM不听话，忽略了上下文 → 优化系统提示词

**解决：**
```python
# 在 config.py 中优化系统提示词
LLM_SYSTEM_PROMPT = """
你是一个专业的问答助手。请严格基于提供的上下文信息回答用户的问题。

要求：
1. 只使用上下文中的信息，不要编造内容
2. 如果上下文中没有答案，直接回答"抱歉，我无法回答这个问题"
3. 回答要简洁、准确，使用中文
4. 不要提及"根据上下文"之类的话，直接给出答案
"""
```

### 问题10: 接口返回 400 Bad Request

**原因：** 上下文太长，超出LLM的上下文窗口限制

**解决：**
```python
# 1. 减小 RETRIEVE_TOP_K，默认10 → 改成5
# 2. 减小传给LLM的文档长度
# 在 routes/ask.py 中截断每个文档的内容
# 或者增加 llama.cpp 启动时的 -c 参数，从 1024 改成 4096
```

## 💾 知识库问题

### 问题11: 新增的知识检索不到

**原因：**
1. 数据没有正确写入ES
2. 没有生成向量
3. ES索引没有刷新

**解决：**
```bash
# 强制刷新索引
curl -k -u elastic:密码 -X POST "https://192.168.1.3:9200/cs_knowledge_base/_refresh"

# 检查文档是否有embedding字段
# 如果没有，重新跑向量生成脚本
```

## 🔄 服务重启顺序

**正确的重启顺序（很重要！）：

```bash
# 1. 先停止所有服务
bash scripts/stop_all.sh

# 2. 等5秒，确保进程都杀干净
sleep 5

# 3. 启动模型服务
/home/zbs/llama_serve.sh

# 4. 等15秒，模型加载完成
sleep 15

# 5. 启动API服务
bash scripts/start_all.sh
```

⚠️ 不要直接 kill -9 然后立刻重启，容易出现端口占用问题。

## 🚨 紧急恢复

如果系统完全崩溃，按以下顺序恢复：

```bash
# 1. 杀所有相关进程
pkill -9 llama-server
pkill -9 python
pkill -9 uvicorn

# 2. 清理显存（如果有进程卡住）
sudo nvidia-smi --gpu-reset

# 3. 重启服务器（终极手段）
sudo reboot
```
