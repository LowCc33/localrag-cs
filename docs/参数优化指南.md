# 参数优化指南

根据你的硬件配置和业务需求，调整以下参数可以获得最佳的性能和效果平衡。

---

## 🎯 检索参数优化

### 1. RETRIEVE_TOP_K - 混合检索召回数量

```python
RETRIEVE_TOP_K = 10  # 默认值
```

| 配置 | 效果 | 适用场景 |
|------|------|---------|
| 5 | 检索快，召回少 | 知识库内容少，对速度要求高 |
| **10** | **平衡** | **大多数场景推荐** |
| 20 | 召回全，但慢 | 知识库内容多，需要高召回率 |
| 50 | 最全，最慢 | 法律、医疗等对召回要求极高的场景 |

✅ **建议：** 先从10开始，测试召回效果后再调整

---

### 2. RERANKER_TOP_K - 重排后传给LLM的数量

```python
RERANKER_TOP_K = 3  # 默认值
```

| 配置 | 效果 | 适用场景 |
|------|------|---------|
| 1 | 回答快，上下文短 | 简单问答，答案唯一 |
| **3** | **平衡** | **大多数场景推荐** |
| 5 | 上下文全，回答更准 | 复杂问答，需要多个文档佐证 |

✅ **建议：** 不要超过5，否则LLM容易"注意力分散"

---

### 3. RRF_K - 混合检索融合参数

```python
RRF_K = 60  # 默认值
```

RRF公式：`score = 1/(k + rank)`

| k值 | 效果 |
|------|------|
| 20 | BM25权重高 |
| **60** | **平衡** |
| 100 | 向量检索权重高 |

✅ **建议：** 60是行业通用值，不需要改

---

## 🧠 LLM 生成参数优化

### 4. LLM_TEMPERATURE - 温度参数

```python
LLM_TEMPERATURE = 0.1  # 默认值
```

| 温度 | 效果 | 适用场景 |
|------|------|---------|
| 0.0 | 最保守，只选概率最高的token | 知识库问答，要求准确 |
| **0.1** | **轻微随机性，兼顾准确和自然** | **客服问答推荐** |
| 0.5 | 有一定创造性 | 闲聊、创意类 |
| 1.0 | 随机性大 | 不推荐用于问答 |

✅ **建议：** 客服场景 0.0-0.2 之间

---

### 5. LLM_MAX_TOKENS - 最大生成长度

```python
LLM_MAX_TOKENS = 1024  # 默认值
```

| 长度 | 效果 | 适用场景 |
|------|------|---------|
| 256 | 回答简洁，快 | 简单FAQ |
| **1024** | **平衡** | **常规问答推荐** |
| 2048 | 回答详细，慢 | 复杂问题，需要长篇解释 |

✅ **建议：** 根据业务需要调整，长回答耗显存

---

## 🚀 llama.cpp 服务参数优化

### 6. --n-gpu-layers - GPU offload层数

```bash
--n-gpu-layers 99  # 把所有层都放到GPU
```

| 配置 | 显存占用 | 速度 |
|------|---------|------|
| 0 | ~1GB CPU内存 | 极慢（几秒/token） |
| 30 | ~3GB | 一般（~10token/s） |
| **99** | **~8GB** | **最快（30+t/s）** |

✅ **建议：** 显存够就设99，把所有层都放到GPU

---

### 7. -c - 上下文窗口大小

```bash
-c 2048  # 默认可能是1024
```

| 窗口大小 | 显存占用 | 支持的上下文长度 |
|------|---------|------|
| 1024 | 基准 | ~700字中文 |
| **2048** | +10% | **~1500字中文** |
| 4096 | +30% | ~3000字中文 |

✅ **建议：** 至少2048，否则问答时很容易OOM

---

### 8. -b - 批处理大小

```bash
-b 512  # 默认
```

对单用户场景影响不大，主要影响并发。

## 🔧 常见调优场景

### 场景1: 我要最快的回答速度

```python
# config.py
RETRIEVE_TOP_K = 5
RERANKER_TOP_K = 1
LLM_MAX_TOKENS = 256
```

```bash

--n-gpu-layers 99

```

**预期：** 2-3秒出答案

---

```python
# config.py

RERANKER_TOP_K = 5
LLM_MAX_TOKENS = 1024
LLM_TEMPERATURE = 0.0
```

```bash
# llama.cpp
--n-gpu-layers 99
-c 4096
```

**预期：** 5-10秒出答案，准确率最高

---

### 场景3: 我显存只有4GB，想跑7B模型

```bash

-c 1024

# 用更小的量化版本 Q3_K_M 代替 Q4_K_M
# 体积小15%，精度下降不明显
```

## 📊 性能监控

### 查看实时推理速度

看LLM服务日志中的输出：
```
prompt_eval_time = 123.45 ms / 10 tokens = 12.345 ms per token
eval_time = 1000.00 ms / 50 tokens = 20.00 ms per token
```

- **prompt_eval:** 理解上下文的速度
- **eval:** 生成回答的速度（这个是关键）

目标：**eval < 50ms/token** 就是流畅的体验

## ⚠️ 注意事项

1. **修改 llama.cpp 参数需要重启服务**
   ```bash
   bash scripts/stop_all.sh
   bash scripts/start_all.sh
   ```

2. **参数不是越大越好**
   - 太大的上下文窗口会显著增加显存占用
   - 太大的TOP_K会让LLM困惑，反而降低质量

3. **先测单用户性能，再测并发**
   - 先确保单个用户体验达标
   - 再逐步增加并发用户

4. **定期监控显存**
   ```bash
   watch -n 1 nvidia-smi
   ```
   - 显存占用超过90%很容易OOM
   - 留10-20%的显存余量

## 🎯 最终建议

**大多数场景用默认配置就很好：**

```python
# config.py
RETRIEVE_TOP_K = 10
RERANKER_TOP_K = 3
RRF_K = 60
LLM_TEMPERATURE = 0.1
LLM_MAX_TOKENS = 1024
```

```bash
# llama_serve.sh
--n-gpu-layers 99
-c 2048
```

这个配置在 8GB 显存的 3090 上可以做到：
- 响应时间：3-5秒
- 生成速度：20-30 token/s
- 显存占用：7-8GB

满足90%的业务场景需求。
